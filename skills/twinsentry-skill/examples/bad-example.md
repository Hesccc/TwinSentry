# 糟糕使用示例 (Bad Example)

本示例展示了在接入 TwinSentry 时应该**尽量避免**的智能体日志输出排版与操作方式。不良的输出格式会导致平台终端的各种安全运营人员阅读困难，并大幅降低故障的阅读和处置效率。

## 场景：检测到 Web 注入攻击 (SQLi)

### 1. 糟糕的分析结论 (Analysis Log)

如果您向 TwinSentry 提交了如下的纯文本作为分析结论：

```text
这可能是一个sql注入，源ip是107.150.119.229，系统提示了很多次报错，不知道是不是真的被黑了，建议你们赶紧看看。
{"source": "waf", "type": "sqli", "confidence": 0.8, "recommendation": "block"}
```

*(缺点：)*
- *缺乏起码的段落排版，通篇纯文本堆砌。*
- *直接暴露了本该作为参数传给对象存储 (`enrichment_data`) 字段的冗余 JSON 结构，污染了前端视图。*
- *语义含糊其辞，未给出确切的定性（威胁级别）和清晰可落地的处理步骤。*

### 2. 糟糕的处置结论 (Action Log)

如果处置 Agent 直接把下发脚本的结果回调回去：

```text
执行执行脚本成功。返回状态码: 200。
调用 waf/api/v1/block 接口 response: {"code":0, "msg":"success", "data":{"banned_ip":"107.150.119.229"}}
结束。
```
*(缺点：)*
- *这完全是针对研发调试人员的打印级别输出，暴露了过多的 API 原始原始响应字段。*
- *未向非技术的运营值班人员提供“人类可读”的处理结果描述（例如简单明了的一句“已对该 IP 完成封禁”）。*
- *忽略了 TwinSentry 对 Markdown 富文本样式的渲染红利，严重拉低了系统的专业感。*
